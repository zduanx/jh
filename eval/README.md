# Chat Agent Evaluation (Phase 7E)

Offline evaluation of the chat agent — measures **faithfulness** (is the answer
grounded in the retrieved data, or hallucinated?) plus relevancy, warmth, and
conciseness. Runs as a test suite, **NOT** in the live request path.

Concept deep-dive: [docs/learning/vectors-rag-eval.md](../docs/learning/vectors-rag-eval.md).
Phase summary: [docs/logs/PHASE_7E_SUMMARY.md](../docs/logs/PHASE_7E_SUMMARY.md).

---

## How it works — two steps

```
1. CAPTURE (Node)   node eval/capture.js
   runs the REAL agent on each test case → writes eval/cases.captured.json
   {question, answer, contexts}   (contexts = the tool results the agent saw)

2. GRADE (Python)   reads cases.captured.json, scores it
   - run_eval.py        hand-rolled LLM-as-judge (no deps but the anthropic SDK)
   - run_eval_ragas.py  Ragas (the named tool) — cross-validates the hand-rolled one
```

Capture is split from grade on purpose: **capture needs the live agent + MCP
server; grade needs only the JSON file + a judge API** — so you capture once and
re-grade many times.

---

## Prerequisites

- **MCP server running** (capture only): `jbemcp-bg` (port 8001).
- **Keys in `chat/.env.local`**: `ANTHROPIC_API_KEY` (the agent + the judge),
  `MCP_SERVER_URL`, `MCP_SERVICE_TOKEN`. The Ragas runner also needs
  `VOYAGE_API_KEY` (in `backend/.env.local`) for the embeddings metric.
- **A user with embedded data.** `capture.js` runs as `EVAL_UID` (default 1).
  The **local** MCP server uses the test DB where **user 18** has the embedded
  jobs — so locally, capture with `EVAL_UID=18`.

## Run it

### 1. Capture (needs MCP server + keys)
```bash
source dev.sh
jbemcp-bg                              # start MCP server, wait ~3s
EVAL_UID=18 node eval/capture.js       # → writes eval/cases.captured.json
jkill-bemcp                            # stop it when done
```
Output per case: `• topic-backend … ok (1 ctx, 479 chars)` — `ctx` = number of
tool calls captured.

### 2a. Grade — hand-rolled judge (lightest)
```bash
source backend/venv/bin/activate
export ANTHROPIC_API_KEY=$(grep '^ANTHROPIC_API_KEY=' chat/.env.local | cut -d= -f2-)
python eval/run_eval.py
```
Scores faithfulness (per-claim) + LLM-judged refusal checks + an adversarial
self-test. Exits non-zero on failure (CI-friendly). Needs only `anthropic`
(`pip install -r eval/requirements.txt`).

### 2b. Grade — Ragas (the named tool, cross-check)
Ragas 0.4.x is broken against current langchain, so it lives in an **isolated venv**:
```bash
# one-time setup:
python3 -m venv eval/.venv-ragas
eval/.venv-ragas/bin/pip install ragas==0.2.10 langchain-community==0.3.3 \
    langchain-anthropic langchain-voyageai datasets

# run:
export ANTHROPIC_API_KEY=$(grep '^ANTHROPIC_API_KEY=' chat/.env.local | cut -d= -f2-)
export VOYAGE_API_KEY=$(grep '^VOYAGE_API_KEY=' backend/.env.local | cut -d= -f2-)
eval/.venv-ragas/bin/python eval/run_eval_ragas.py
```
Scores faithfulness (customized to ignore conversational filler) + answer_relevancy
(Voyage embeddings) + warmth & conciseness (AspectCritic). Uses **Anthropic** as
the judge, not OpenAI.

## Files

| File | What |
|------|------|
| `cases.js` | the test set — questions, `uid`, optional `history`, `expectRefusal` |
| `adversarial.json` | hand-authored known-bad case (must score LOW — proves the detector bites) |
| `capture.js` | runs the real agent, records `{question, answer, contexts}` |
| `run_eval.py` | hand-rolled LLM-as-judge grader |
| `run_eval_ragas.py` | Ragas grader (cross-validation) |
| `cases.captured.json` | generated by capture (gitignored) |
| `.venv-ragas/` | isolated venv for Ragas (gitignored) |

## Adding a test case

Edit `cases.js`:
```js
{ id: 'my-case', question: 'show me my ML jobs' }            // graded for faithfulness
{ id: 'no-data', question: '...', expectRefusal: true }      // should DECLINE, not fabricate
{ id: 'multi',  question: '...', history: [ {role,content} ] } // multi-turn
```
Then re-run **capture** (to produce fresh answers) before grading.

To test that the detector catches hallucinations, add a known-bad case to
`adversarial.json` (a hand-written answer with claims NOT in its `contexts`).

## Interpreting scores

- **faithfulness** = supported claims / total claims (per case), averaged. **Hard
  gate** (lying is unacceptable). 1.0 = every factual claim was grounded.
- **answer_relevancy** = does the answer address the question (embeddings). Tracked.
- **warmth / conciseness** = AspectCritic yes/no over the whole answer → **pass-rate**.
  **Tracked signals, not gates** — surface for human judgment (e.g. a terse
  follow-up correctly scores not-warm; that may be fine).
- Treat scores as a **regression signal**, not absolute truth. **Pin the judge
  model** (`EVAL_JUDGE_MODEL`); changing it recalibrates the scale. Thresholds are
  calibrated to baseline, not a magic 0.85.
