# Vectors, RAG & LLM Evaluation

**Context**: Concepts behind semantic job-matching and chat-agent evaluation
(Phase 7). Covers embeddings, ANN indexes (HNSW/IVFFlat), RAG, and eval
frameworks (Ragas/DeepEval).

---

## Embeddings (vectors)

A **vector / embedding** is a list of numbers (e.g. 384–1536 dims) representing
the *meaning* of text as a point in high-dimensional space. Similar meaning →
vectors that are close together.

- Generated by a dedicated **embedding model** (NOT a chat model). Claude/GPT
  generate *text*; embedding models (OpenAI `text-embedding-3`, Voyage, Cohere,
  open-source `sentence-transformers`) generate *vectors*. You call the embedding
  model; you don't ask Claude for a vector.
- **Comparison = cosine similarity**: `(A·B)/(|A||B|)`, range −1..1. 1 = same
  meaning, 0 = unrelated. Measures the *angle* between vectors. Done in
  code/the DB — **the LLM never compares vectors**.

```
text → [embedding model] → vector
"distributed systems"  ≈  "large-scale infrastructure"   (close vectors; keyword search misses this)
```

## ANN, HNSW, IVFFlat — fast vector search

Comparing a query vector against *every* stored vector is O(n) — slow at scale.
**Approximate Nearest Neighbor (ANN)** indexes find the closest vectors quickly
without checking all of them (trading a little accuracy for big speed).

- **HNSW** (Hierarchical Navigable Small World) — a layered **graph** index.
  Navigates from coarse to fine to find nearest neighbors fast. Fast queries,
  higher memory. The common default.
- **IVFFlat** (Inverted File) — **partitions the vector space into clusters**; a
  query only searches the nearest few clusters, not all vectors. Lower memory,
  needs a "train" step, slightly less accurate. A form of partitioning *by
  similarity*.
- **Exact (flat)** — compare against all; fine for small datasets (hundreds).

At small scale (e.g. 300 jobs) any of these is sub-millisecond; the index matters
only as the corpus grows.

### Partitioning / sharding vectors
- **By similarity** (IVFFlat clusters) — search fewer clusters.
- **By metadata** (per-user/tenant/time) — filter to one tenant's vectors (e.g.
  `WHERE user_id = ...`).
- **Sharding across nodes** — only at billions-of-vectors scale (dedicated vector
  DBs). For this project: single HNSW index + `WHERE user_id` filter. No sharding.

## pgvector vs a dedicated vector DB

**`pgvector`** is a Postgres extension adding a `vector` column type, similarity
operators (`<=>` cosine distance), and ANN indexes (HNSW/IVFFlat) — turning
Postgres into a vector DB.

```sql
CREATE EXTENSION vector;
ALTER TABLE jobs ADD COLUMN embedding vector(1536);
SELECT id, title FROM jobs WHERE user_id = :uid
ORDER BY embedding <=> :query_vector LIMIT 10;   -- nearest 10 by cosine
```

**Use pgvector (this project) when:** you already run Postgres, modest scale,
want vectors co-located with relational data (filter + vector search in one
query), and want to avoid a second datastore. (Same philosophy as ADR-014's
"PostgreSQL-native search, no external services.")

**Use a dedicated vector DB (Pinecone/Milvus/Weaviate) when:** tens of millions+
vectors, very high throughput, cross-node sharding, specialized features. Not
needed here.

## RAG (Retrieval-Augmented Generation)

The pattern: **Retrieve** relevant info → **Augment** the prompt with it →
**Generate** a grounded answer.

```
query → RETRIEVE relevant docs → put them in the prompt → LLM GENERATES grounded answer
```

- The "Retrieval" can be **vectors** (semantic), **keyword** (tsvector/pg_trgm),
  or **SQL filters** — RAG ≠ vectors necessarily. It's RAG once retrieved results
  feed the LLM.
- **Why:** a corpus too large for the context window (e.g. 300 JDs) — retrieve
  the top-K relevant, let the LLM reason over just those.
- **Division of labor:** vectors/DB are a fast **FILTER** (300 → top-10, in
  code); the LLM is the nuanced **JUDGE** over the filtered set (text, not
  vectors). Sequential, not "LLM compares vectors."

**This project:**
- Broad query ("best of my 300 jobs for my resume") → **RAG** (pgvector retrieve
  top-K → LLM ranks/explains).
- Single comparison ("how do I fit THIS job") → **no retrieval**; inline that JD +
  resume directly (grounded generation, not really RAG).

## LLM Evaluation — Ragas, LLM-as-Judge (hands-on, Phase 7E)

Turn "does the answer seem good?" into **measurable metrics**. Run **offline / in a
test suite**, NOT in the live request path. (Built in 7E — see [PHASE_7E_SUMMARY](../logs/PHASE_7E_SUMMARY.md).)

### The capture → grade split (and why)
The eval has two steps, decoupled on purpose:
```
capture: run the REAL agent on each test question → record {question, answer, contexts}
grade:   feed those triples to a judge → scores
```
- **Capture needs the live agent** (+ MCP server). **Grade needs only the data file +
  judge API** — so you capture once, re-grade many times (e.g. while tuning the judge)
  without re-running the agent. Industry does this via **tracing** (LangSmith/Langfuse);
  our scaled-down version writes `eval/cases.captured.json`.
- **Contexts ≠ the answer.** The answer comes from the agent's output stream; the
  **contexts (tool results) must be captured by instrumenting the agent** — we *wrap
  `tools.call` externally* (decorator/monkey-patch) before passing tools in, so every
  tool result is recorded as a side-effect. Possible only because tools are
  *dependency-injected*. "What you capture determines what you can grade."
- **Same-language note:** Ragas is Python; our agent is Node, so we *must* bridge via a
  JSON file. If the agent were Python, you'd typically call it **in-process** in the
  eval script and skip the file. Ragas always grades a **dataset of rows**, never your
  backend directly — the only question is whether you produce those rows in-process or
  via a file.

### How faithfulness actually works (decompose → judge → ratio)
The headline metric. NOT a single fuzzy "rate 0–1" — it's **arithmetic over binary
judgments**, which is what makes it stable:
```
1. SLICE  (LLM): break the answer into atomic factual claims      ← extraction prompt
2. JUDGE  (LLM): for each claim, supported by context? 1/0         ← NLI prompt
3. SCORE  (code): faithfulness = supported / total                ← plain division
```
- **The LLM judges; code computes.** Never ask the LLM for the final number — ask it
  many narrow yes/no questions and *count* them. (Decision: low temperature on the
  judge for reproducibility.)
- **Granularity = claim count.** Score resolution comes from *how many claims*, not LLM
  "confidence." 3 claims → score ∈ {0, .33, .67, 1}. A clean **1.0 means every claim
  checked out**; you only see 0.98 with ~50 claims and 1 miss. Small clean cases → exact
  1.0, not 0.98.
- **Ragas vs. hand-rolled:** *identical algorithm.* Ragas uses **2 LLM calls** (slice,
  then judge); our hand-rolled judge does **1 combined call** (slice+judge in one
  prompt). Both return claims+verdicts → ratio. They agreed at 1.0 — cross-validation.
- **The claims are LLM-generated, not pre-defined** — step 1 atomizes the prose
  ("3 OpenAI jobs" + "all in SF" → "all 3 OpenAI jobs are in SF"). Smart, but it also
  turns *filler* ("Want me to dive deeper?") into a fake "claim" → see below.

### LLM-as-judge: why "AI grading AI" is OK (and its limits)
- **It's circular but defensible:** judging a narrow claim ("is X in the context?") is
  far easier + more verifiable than generating, and it's the only scalable way to grade
  free-text. Treat scores as a **regression signal, not truth** — track trends, alert on
  drops, set thresholds with margin (fail < 0.85, not "== 0.93").
- **Pin the judge model (it's the measuring instrument).** Two models are in play: the
  **agent** (under test — vary freely) and the **judge** (grades — keep fixed). Changing
  the judge recalibrates the scale and breaks comparability → re-baseline if you must.
- **No universal threshold.** "0.85" is not an industry standard — calibrate to your
  system's measured baseline and domain stakes (high-stakes → 0.9+, low-stakes → 0.7).
  A threshold above your known-good baseline is a misconfigured check.

### Multi-dimensional eval (one answer, many axes)
"Good answer" isn't one thing → measure separate dimensions, each its own judge:
- **faithfulness** (facts grounded) — per-*claim* ratio.
- **answer_relevancy** (addresses the question) — uses **embeddings** (Voyage here):
  embeds the question vs. a question regenerated from the answer, cosine similarity.
- **warmth / conciseness** (tone) — Ragas **AspectCritic** = a yes/no rubric judge over
  the *whole answer*; the reported number is a **pass-rate** (fraction of cases = 1),
  not a per-answer partial score.
- **Different enforcement per dimension:** faithfulness is a **hard gate** (lying is
  unacceptable); warmth/conciseness are **tracked signals** (don't fail a build for a
  slightly-terse answer — surface it for human judgment).

### The warmth ↔ faithfulness tension (a real production problem)
- Off-the-shelf faithfulness **penalizes conversational warmth**: it atomizes "Let me
  check!" / "Want me to dive deeper?" into "claims" → marks them unsupported → low score
  on a *perfectly grounded* answer. That's the metric punishing *style*, not fabrication.
- **The fix is NOT to make the agent robotic.** The metric is a *proxy*; optimizing it
  until it hurts UX is Goodhart's law. Instead, **fix the metric** (Approach #1):
  customize the extraction prompt to *exclude conversational framing* so filler never
  becomes a claim. This took our Ragas faithfulness 0.76 → 1.0 **without touching the
  agent** — and made it agree with the hand-rolled judge.
- **Or measure warmth separately and positively** (Approach #2 — AspectCritic). Then
  warmth *helps its own metric* and is *invisible* to faithfulness. This also closes a
  blind spot: a cold-but-grounded answer scores faithfulness=1.0 and would otherwise look
  perfect — warmth=0 flags it. (In our run, a terse factual follow-up — "The id is 163."
  — correctly scored warmth=0: accurate but transactional, no greeting/offer. Whether
  that's a *defect* is human judgment, not the metric's call.)

### Gotchas we actually hit (debugging the eval is normal)
- **Ragas 0.4.x is broken vs. langchain 1.x** (imports a removed `vertexai` module).
  Pin **ragas 0.2.x + langchain-community 0.3.3** in an **isolated venv**.
- **Use Anthropic, not OpenAI:** Ragas *defaults* to OpenAI but is configurable —
  pass a `LangchainLLMWrapper(ChatAnthropic(...))` judge + `VoyageAIEmbeddings`. (Only
  embedding-based metrics like answer_relevancy need embeddings; faithfulness is LLM-only.)
- **`max_tokens` too low → `LLMDidNotFinishException`** on long answers (Ragas's prompts
  are verbose). Give the judge generous output room (4096).
- **Markdown breaks the segmenter:** Ragas mis-scored a fully-grounded markdown-table
  answer to **0.000** — its sentence-splitter garbled the table. Fix = **normalize
  markdown → plain prose before grading** (separate from the filler fix; one un-breaks
  parsing, the other stops over-counting non-facts).
- **Validate the eval against known cases before trusting numbers.** A broken metric
  (the 0.000) is worse than none. We caught it by **cross-validating two judges** and
  feeding the NLI step clean statements (all ✓) — proving the agent was faithful and the
  0.000 was a tool artifact.
- **Negative test required:** a faithfulness metric that never sees a hallucination is
  unproven. Add an **adversarial case** (a hand-written answer with claims NOT in the
  context) and assert it scores *low* — proves the detector bites.

### Refusals & "expectRefusal"
Some questions should be **declined**, not answered ("tell me about job 163" when 163
doesn't exist; "what's the weather?"). Faithfulness doesn't apply (no claims to ground —
Ragas even scored a refusal 0.000). Route these to a **separate refusal check**:
"did the agent appropriately decline rather than fabricate?" (We replaced a brittle
keyword list with a small **LLM-judge** — prose phrasings are too varied to enumerate.)

**This project:**
- **Chat agent (Phase 7E)** → Ragas + a hand-rolled judge (cross-validated), faithfulness
  + relevancy + warmth/conciseness. Measures the anti-fabrication promise with numbers.
- **Admin agent (Phase 8)** → NOT Ragas. Output is **structured data**, gradable
  **deterministically** (extracted title appears in source HTML, URL well-formed) against
  `trials/{company}/` golden fixtures — plus precision/recall (valid ≠ good). Pydantic
  validates structure; LLM-judge only for fuzzy parts. **Validation (inline gates) vs.
  evaluation (offline quality)** are different disciplines for different output types.

## Semantic caching (and why it's a poor fit here)

Caching LLM responses by **meaning** (embed query → if a cached query is
cosine-similar above a threshold, return its answer; skip the LLM).

- Great for **stateless, shared** queries (support FAQ: "reset password" → same
  answer for everyone).
- **Poor fit for personalized/stateful chat**: "which job fits best?" needs a
  *different* answer per user/session — a semantic cache would return wrong
  answers. Use **exact-match** caching on deterministic tool results instead
  (e.g. `score_against_jd(resume_hash, jd_hash)`), not semantic caching of chat
  answers.

## Key Takeaways

1. **Embeddings come from a separate embedding model, not the chat LLM.** Cosine
   similarity (in code/DB) compares them; the LLM never sees vectors.
2. **RAG = retrieve → augment → generate.** Retrieval can be vector/keyword/SQL.
   Vectors *filter*; the LLM *judges* the filtered set.
3. **pgvector on existing Postgres beats a separate vector DB** at this scale —
   co-located, fewer moving parts.
4. **Eval is offline and measurable.** Ragas faithfulness turns "I don't
   hallucinate" into a number. This is the "developing evaluations" discipline.
5. **Match the technique to the problem** — semantic caching fits shared FAQ, not
   personalized chat. Knowing *when not to* use a technique is the senior signal.
